## 🔄 修正サイクル

1. ドライランで詰まりポイントを洗い出す
2. 「ドキュメント修正が必要な箇所」の重要度でソート
3. 上から順に修正（`docs-site/index.html`、`starter/README.md`、`starter/main.js` など）
4. 修正した箇所を再度動作確認
5. 全て解消したら別の人にもう一度ドライラン依頼

---

## 📊 ドライラン後の判断基準

以下すべてが Yes なら本番 GO：

- [ ] 60 分（+/- 5 分）で本編を完了できた
- [ ] 全 5 章のコアフロー（STEP 1）が動作した
- [ ] クリティカルなドキュメント不明瞭点がない
- [ ] メンターチートシートに載せるべき詰まりポイントを反映した
- [ ] 事前アナウンスの手順で新規環境から `available` × 5 になった

## 🚨 ドライラン中によく起きるパターンと対処

### パターン A: 初回モデル DL に時間がかかりすぎる
→ 事前アナウンスで「モデル DL は 10〜30 分かかることがある」と目安を明記
→ 会場では「モデル DL 済み」を必須条件にする（Chrome 148 以降のインストールも合わせて）

### パターン B: 特定の章で 5 分オーバー
→ 該当章の STEP を精査、削れる STEP や見本コードで先送りできる部分を探す
→ 60 分に厳密にフィットさせるのではなく「順不同で好きな章を深めればいい」ことを再徹底

### パターン C: `docs-site` の説明が「わかりにくい」と言われた
→ どの部分が不明瞭かを具体的に聞き取り、該当箇所に例やイラストを追加
→ API リファレンスの引数/戻り値説明を充実させる

### パターン D: Chrome の API 仕様が変わっていた
→ solutions/main.js と docs-site の該当スニペットを即修正
→ starter/main.js の TODO 例も更新
→ event-materials/mentor-cheatsheet.md にも変更点を追記
